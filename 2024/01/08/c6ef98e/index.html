MySQL-高级知识掌握

数据库概念

  • 数据库: 数据存储的仓库
  • 数据库管理系统：操纵和管理数据库的大型软件
  • SQL操作关系行数据库的编程语言,是一套标准

MySQL数据库

  • 关系型数据库

    • 概念：建立在关系模型基础上,由多张相互连接的二维表组成的数据库
    • 特点
      • 使用表存储数据,格式统一,便于维护
      • 使用SQL语言操作,标准统一,使用方便

流程函数
  • if(value,true,false)如果为true，则返回t，否则返回f
  • ifnull(value1,value2)如果value1不为空,返回value1,否则返回value2
  • case when [val1] then [res1]..else [default] end如果val为true，返回res1，否则返回default默认值
  • case [expr] when [val] then [res1]..else [default] end如果expr的值等于val1，返回res1,否则返回default默认值

多表查询

多表关系

  • 概述

    项目开发中,在进行数据库表结构涉及时,会根据业务需求及业务模块之间的关系,分析并设计表结构,由于业务之间相互关联,所以各个表结构之间也存着各种关系,基本上分为三种

    • 一对多(多对一)
    • 多对对
    • 一对一
多表查询概述

  • 笛卡尔积：笛卡尔积是指在数学中,两个集合A集合和B集合的所有组合情况(在多表查询时,需要消除无效的笛卡尔积)

  • 分析

内连接
  • 相当于查询A、B交集部分的数据
外连接
  • 左外连接：查询左表所有数据，以及两张表交集部分数据
  • 右外连接：查询右表所有数据，以及两张表交集部分数据
自连接
  • 当前表与自身的连接查询,自连接必须使用表别名
子查询

  • 概念

    SQL语句中嵌套select语句,称为嵌套查询,又称子查询

  • 根据子查询结果不同,分为

    • 标量子查询(子查询结果为单个值)

    • 列子查询(子查询结果为一列)

      • 常用操作符：In、Not in、any、some、all

    • 行子查询(子查询结果为一列)

  • 表子查询(子查询结果为多行多列)

事务

  • 概念

    事务是一组操作的集合,他是一个不可分割的工作单位,事务会把所有的操作作为一个整体一起向系统提交或撤销操作请求,即这些操作要么同时成功,要么同时失败

  • 四大特性

    • 原子性：事务是不可分割的最小操作单元,要么全部成功,要么全部失败
    • 一致性：事务完成时,必须使所有的数据都保持一致状态
    • 隔离性：数据库系统提供的隔离机制,保证事务在不受外部并发操作影响的独立环境下运行
    • 持久性：事务一旦提交或回滚,它对数据库中的数据的改变就是永久的

  • 并发事物引发的问题

    • 脏读

    • 不可重复读

    • 幻读

存储引擎

  • InnoDB存储引擎

    • 具备外键支持功能的事务存储引擎
    • MySQL 大于等于 5.5之后,默认采用InnoDB引擎
    • 除了增加和查询外,还需要更新、删除等操作，那么,应优先选择InnoDB引擎

  • 存储引擎选择

    在选择存储引擎时,应根据应用系统的特点选择合适的存储引擎,对于复杂的应用系统,还可以根据实际情况选择多种存储引擎进行组合

    • InnoDB

      是MySQL的默认存储引擎,支持事务、外键、如果应用对事务的完整性有比较高的要求,在并发条件下要求数据的一致性,数据操作除了插入和查询之外,还包括更多的更新和删除操作,那么InnoDB存储引擎是比较合适的选择

    • MyISAM

      如果应用是以读操作和插入操作为主,只有很少的更新和删除操作,并且对事务的完整性,并发性要求不是很高,那么选择这个存储引擎是比较合适的

    • MEMORY

      将所有的数据保存在内存中,访问速度快,通常用于临时表及缓存.MEMORY的缺陷就是对表的大小有限制,太大的表无法缓存在内存中,而且无法保障数据的安全性

  • 索引的优缺点

    • 优点
      • 提高数据检索的效率,降低数据库的IO成本
      • 通过索引列对数据进行排序,降低数据库排序的成本,降低CPU的消耗
    • 缺点
      • 索引列也是要占用空间的
      • 索引大大提高了查询效率,同时却也降低了更新表的速度,如果对表进行insert、update、delete时,效率降低

  • 索引的数据结构: 详细了解了B+Tree 、Hash索引结构

    • 二叉树

      二叉树缺点: 顺序插入时,会形成一个链表,查询性能大大降低,层级较深,检索速度慢

      红黑树: 大数据量情况下,层级较深,检索速度慢

    • B-Tree(多路平衡查找树)

      以一颗最大度数为5阶的b-tree为例(每个节点最多存储4个key，5个指针)

      树的度数指的是一个节点的子节点个数

    • B+Tree

      所有数据都会出现在叶子节点，非叶子节点会起一个索引的作用,叶子节点是一个单项链表结构

      • MySQL中的B+Tree

        MySQL索引数据结构对经典的B+Tree进行了优化,在原B+Tree的基础上,增加了一个指向相邻叶子节点的链表指针,就形成了带有顺序指针的B+Tree,提高区间访问的性能

    • Hash索引特点

      • Hash索引只能用于对等比较(=,in),不支持范围查询(between,>,<)
      • 无法利用索引完成排序操作
      • 查询效率高，通常只需要一次索引就可以了,效率通常要高于B+Tree索引

索引分类

  • 分类

  • 在InnoDB存储引擎中,根据索引的存储形式,又可以分为以下两种

    • 聚集索引选取规则

      • 如果存在主键,主键索引就是聚集索引
      • 如果不存在主键,将使用第一个唯一索引作为聚集索引
      • 如果表没有主键,或没有合适的唯一索引,则InnoDB会自动生成一个rowId作为隐藏的聚集索引

      在二级索引中,叶子节点放的是对应值(name=Kit,...)的id,而聚集索引的叶子节点放的是这一行数据

    explain或者desc命令获取MySQL如何执行Select语句的信息,包括在select语句执行过程中表如何连接和连接的顺序

    • 语法

    • 字段解释说明

      • id: select查询的序列号,表示查询中执行了select子句或者是操作表的顺序(id相同,执行顺序从上到下;id不同,值越大,越先执行)
      • select_type：这个字段表示了查询的类型，可能的值包括：
        • SIMPLE：简单查询，不包含子查询或UNION
        • PRIMARY：查询中的最外层查询。
        • UNION：表示该查询是一个UNION的结果。
        • SUBQUERY：子查询。
        • DERIVED：派生表，即从FROM子句的SELECT列表中获取数据的子查询。
        • UNION RESULT：UNION的结果集。
      • table：指明了查询中涉及的表名。
      • partitions：如果使用了分区表，这个字段会显示使用的分区。
      • type：表示了连接类型或访问类型。常见的值包括：
        • const：使用常量值进行连接，通常用于在索引中查找唯一行。
        • eq_ref：在连接中使用了索引，并且索引是唯一索引。
        • ref：在连接中使用了普通索引。
        • range：使用了索引的范围扫描。
        • index：全表扫描，但只使用了索引中的信息。
        • all：全表扫描。
      • possible_keys：指明了查询中可能使用到的索引。
      • key：指明了实际使用到的索引。
      • key_len：表示索引中使用的字节数。
      • ref：显示了使用索引的列或常量值。
      • rows：表示了MySQL估计查询中需要检查的行数。
      • filtered：表示了行的过滤率，即查询条件过滤后剩余的行数占总行数的百分比。
      • Extra：提供了关于查询执行的其他信息，如使用了临时表、文件排序等。

为什么InnoDB存储引擎选择使用B+Tree索引结构

  • 先讨论二叉树,再到二叉树的解决方案,提出红黑树,但是红黑树也属于二叉树的一种,没有脱离二叉树的缺陷
  • 对于B-Tree,无论是叶子节点还是非叶子节点,都会保存数据,这样会导致一页中存储的键值减少，指针跟着减少,要同样保存大量数据,只能增加树的高度,导致性能降低,B-Tree的叶子节点是一个单项链表,不利于范围查询,B+Tree是一个双向链表,有利于范围搜索和排序
  • 关于没有采用Hash索引是因为Hash索引由Hash索引只支持等值,不支持范围查询
